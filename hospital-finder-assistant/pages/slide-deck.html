Hospital Finder Assistant

A conversational assistant that finds hospitals by location, specialty and insurance — through natural voice or text. Speech → intent → semantic retrieval → grounded response → speech.

The Brief

The problem

Build an AI voice bot that helps a user find the right network hospital — hands-free, conversational, and aware of where they are, what specialty they need, and which insurance they carry.

Must handle

  • Spoken & typed queries
  • Messy, partial input
  • Missing info (ask follow-ups)
  • Natural spoken replies

My take

  • End-to-end, runnable system
  • Realistic synthetic UAE dataset
  • RAG for grounded answers
  • A fine-tuned model in the loop

Every step is a node in a LangGraph state machine. State carries the audio paths, transcription, recognition, retrieved hospitals and final response across turns — so the bot can loop, clarify, and continue naturally.

Understanding

Six intents, one recognizer

Intents

  • find_nearest — by distance
  • find_best — by rating
  • find_by_insurance — by provider
  • find_by_hospital — specific facility
  • compare_hospitals — side by side
  • exit — end chat

Entity extraction

  • Location → geocoded to lat/long
  • Specialty (≈50 types)
  • Insurance provider (≈25)
  • Hospital names, count, radius

Two engines: an LLM recognizer (Gemini 2.0 Flash, structured JSON) or spaCy NER + RapidFuzz fuzzy matching.

Retrieval Augmented Generation

Grounding the answer

Hospitals are embedded into a FAISS vector store. A query is built from the user's location, specialty and insurance, then:

  • Semantic search pulls the closest candidates (plus a few extras for context)
  • Haversine distance filters by the requested radius
  • Results are sorted by distance (nearest) or rating (best)
  • An LLM grounds the shortlist into a concise, speakable recommendation — selecting the exact hospital IDs and writing the dialogue
Fine-tuning

A QLoRA model in the loop

For find_by_hospital queries, grounding switches to a locally fine-tuned model — trained on synthetic hospital/insurance dialogue.

  • Base: Open-LLaMA-3B v2
  • 4-bit quantization (NF4, bitsandbytes)
  • LoRA adapters on q_proj / v_proj
  • PEFT + HF Trainer, 3 epochs

Why

  • Cheap, on-device specialization
  • Domain-tuned phrasing
  • No per-call API cost for the specialized path
  • Demonstrates the full train → adapt → serve loop
